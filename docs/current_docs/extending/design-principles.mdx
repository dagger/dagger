# Designing Great Modules

A great module feels smaller than the problem it solves. It gives callers names that match their intent, defaults that match the project, errors that explain recovery, and outputs that compose naturally with Dagger.

## Design From Calls Backward

Start with the call you want people to remember:

- `dagger call test`
- `dagger call image publish`
- `dagger check`
- `dagger generate`

Then design the module to make that call obvious.

Good function names are verbs or nouns with a clear result. Good object names reflect durable concepts in the user's domain, not implementation details.

## Keep Inputs Honest

Every input should answer one question for the caller. Avoid "options bag" arguments that hide many concepts in one string.

Prefer:

- Specific types over strings when Dagger has a type.
- Defaults for project conventions.
- Settings for workspace-level defaults.
- Secrets for credentials.
- Directories and files for host or Git content.
- Enums when only a known set of values is valid.

Avoid:

- Magic environment variables.
- Host path assumptions.
- Flags whose meaning changes by context.
- Inputs that are required only because the module did not set a reasonable default.

## Return Composable Values

Return values should help the caller continue the workflow. A function that builds an image may return a `Container`; a function that generates code may return a `Changeset`; a function that computes metadata may return a structured object.

Avoid flattening everything into a string just because it is easy to print.

## Make Side Effects Explicit

Dagger functions are easiest to reason about when their effects are visible in their return values.

Use:

- `Changeset` for proposed workspace edits.
- `Service` for long-running network dependencies.
- `Secret` for sensitive inputs.
- `File` and `Directory` for artifacts.
- Explicit publish or export functions for external side effects.

## Progressive Disclosure

A module should be useful at first glance and deep when needed.

The first layer is:

- One or two obvious top-level functions.
- Helpful `dagger functions` output.
- Clear defaults.

The second layer is:

- Object methods for advanced composition.
- Settings for team defaults.
- Checks and generators for automation.
- Good troubleshooting docs.

## API Review Questions

Use these questions before publishing a module:

- Can a new caller guess the main function name?
- Does each required input represent something the caller genuinely must decide?
- Are project conventions defaults or settings instead of repeated arguments?
- Are secrets, services, directories, files, and side effects explicit?
- Does each function return the richest useful Dagger value instead of flattening into text?
- Would the trace read like the workflow the caller thinks they ran?
- Is the first failure likely to be actionable?
- Can another module compose this output without parsing logs?
