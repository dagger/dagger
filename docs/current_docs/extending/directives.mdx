# Directives

Directives and metadata are how module authors teach Dagger, tools, and callers more about an API than a raw function signature can express.

They influence discoverability, default behavior, checks, generators, documentation, and source handling.

## Documentation Metadata

Every public module, object, function, argument, and important return value should have documentation.

Good documentation metadata:

- Starts with the caller's goal.
- Names side effects.
- Explains defaults.
- States what a returned object can be used for.
- Gives recovery guidance for common failures.

Bad documentation metadata repeats the function name in a sentence.

## Metadata Is Product Design

Metadata is part of the module's user interface. It shapes what callers see in help output, generated clients, traces, checks, and source change previews.

Treat metadata as the place to answer questions that a type signature cannot:

- Why would I call this?
- What happens if I omit this argument?
- Which files does this default path read?
- Is this function safe to run in CI?
- Does this function propose changes or perform an external side effect?
- What should I do when it fails?

## Default Paths and Workspace Inputs

Default paths are one of the most important module authoring tools. They let a function say, "If the caller does not pass a directory, use this part of the workspace."

Use them to make common calls short, but be careful with scope:

- Default to the smallest useful directory.
- Avoid defaulting to the entire workspace unless the function truly needs it.
- Pair broad defaults with ignore patterns.
- Explain defaults in argument docs.

## Checks and Generators

Checks and generators turn module functions into workspace automation.

Use checks for repeatable project health signals:

- Tests.
- Linters.
- Security scans.
- Policy checks.
- Documentation checks.

Use generators for proposed source changes:

- Generated clients.
- Formatted code.
- Lockfile updates.
- Boilerplate or configuration changes.
