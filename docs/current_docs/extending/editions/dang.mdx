# Developing Modules — Base Edition

This guide uses **Dang**, Dagger's native DSL. It maps directly to the Dagger API with no codegen — what you write is what runs. No build step, no generated files, no runtime overhead.

Use Dang when your module is primarily orchestrating containers, files, and other Dagger objects. If you need external libraries (a Go parser, a Python ML library, a Node.js bundler API), use the [Go](/extending/editions/go), [TypeScript](/extending/editions/typescript), or [Python](/extending/editions/python) edition instead.

## Project setup

```shell
dagger module init --sdk=dang my-module
```

This creates a module at `.dagger/modules/my-module/` with a single `main.dang` file.

## Language basics

Dang is intentionally small:

- **Types** are declared with `type Name { ... }`
- **Public members** use `pub`, private use `let`
- **Functions** are type members that return a value: `pub build: Container! { ... }`
- **Arguments** go in parentheses: `pub build(source: Directory!): Container! { ... }`
- **Annotations** modify behavior: `@check`, `@generate`, `@defaultPath`, `@ignorePatterns`
- **Non-null** is marked with `!`, nullable is the default
- **Descriptions** use triple-quoted strings: `""" ... """`
- **Comments** use `#`
- **Module metadata** uses top-level `pub`: `pub description = "..."`

## Your first module

A module is a type with public functions:

```dang
pub description = "CI for my project"

type MyCi {
  pub hello: String! {
    "Hello from Dagger!"
  }
}
```

`pub` makes things visible to callers. `description` is the module's one-line summary shown in `dagger functions` and the Daggerverse.

Try it:

```shell
dagger call hello
# Hello from Dagger!
```

## Working with containers

Most modules orchestrate containers. The Dagger API gives you a fluent builder:

```dang
type MyCi {
  pub build: Container! {
    container
      .from("node:20")
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["npm", "install"])
      .withExec(["npm", "run", "build"])
  }
}
```

Each method returns a new immutable container — nothing mutates. Dagger caches each step automatically, so re-running skips unchanged work.

## Accepting arguments

Functions accept typed arguments:

```dang
type MyCi {
  pub build(
    """
    Node.js version to use
    """
    nodeVersion: String! = "20",
  ): Container! {
    container
      .from("node:" + nodeVersion)
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["npm", "install"])
      .withExec(["npm", "run", "build"])
  }
}
```

Arguments can have:
- **Types**: `String!`, `Int!`, `Boolean!`, `Directory!`, `File!`, `Secret`, `Container!`, etc.
- **Defaults**: `= "20"` — makes the argument optional
- **Descriptions**: Triple-quoted strings above the argument
- **Non-null markers**: `!` means required (when no default), absent means nullable

```shell
dagger call build
dagger call build --node-version=18
```

## Accessing the workspace

Modules access the project's files through directory arguments with `@defaultPath`:

```dang
type MyCi {
  let source: Directory! @defaultPath(path: "/") @ignorePatterns(patterns: [
    "node_modules",
    ".git",
    "dist",
  ])

  pub build: Container! {
    container
      .from("node:20")
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["npm", "install"])
      .withExec(["npm", "run", "build"])
  }
}
```

`@defaultPath(path: "/")` means "use the workspace root when no directory is explicitly passed." `@ignorePatterns` filters out files that aren't needed — this is important for cache efficiency. The less you load, the fewer cache invalidations.

**Principle: push file access to the leaves.** Don't load the entire repo if you only need `src/`. Use specific paths and tight ignore patterns.

```dang
type MyCi {
  # Only load what we need
  let source: Directory! @defaultPath(path: "/src")
  let config: File! @defaultPath(path: "/tsconfig.json")
}
```

## Private state with `let`

`let` defines private bindings — internal state and helpers not exposed to callers:

```dang
type Security {
  # Private: not callable by users
  let trivyBase = container
    .from("aquasec/trivy:0.68.2")
    .withMountedCache(
      path: "/root/.cache",
      cache: cacheVolume("trivy-cache"),
      sharing: CacheSharingMode.LOCKED,
    )
    .withWorkdir("/home/trivy")

  # Public: callable by users
  pub scanSource(source: Directory! @defaultPath(path: "/")): Void {
    trivyBase
      .withMountedDirectory(".", source)
      .withExec(["trivy", "fs", "--exit-code=1", "--severity=CRITICAL,HIGH", "."])
      .sync
    null
  }
}
```

`let` bindings are computed lazily and cached. Use them for shared setup that multiple functions reuse.

## Checks

A check is a function that validates something without requiring arguments. Mark it with `@check`:

```dang
type MyCi {
  let source: Directory! @defaultPath(path: "/")

  """
  Lint the code
  """
  pub lint: Void @check {
    container
      .from("golangci/golangci-lint:latest")
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["golangci-lint", "run"])
      .sync
    null
  }

  """
  Run unit tests
  """
  pub test: Void @check {
    container
      .from("golang:1.22")
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["go", "test", "./..."])
      .sync
    null
  }
}
```

Now `dagger check` discovers and runs both. A check passes if it completes without error; it fails if any `withExec` returns a non-zero exit code.

Checks can also return `Container` — Dagger will sync it and use the exit code:

```dang
pub lint: Container @check {
  container
    .from("golangci/golangci-lint:latest")
    .withDirectory("/app", source)
    .withWorkdir("/app")
    .withExec(["golangci-lint", "run"])
}
```

## Generators

A generator produces a changeset — a diff between the current source and generated output. Mark it with `@generate`:

```dang
type MyCi {
  let source: Directory! @defaultPath(path: "/")

  """
  Generate Go code from protobuf definitions
  """
  pub generateProto: Changeset @generate {
    container
      .from("bufbuild/buf:latest")
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["buf", "generate"])
      .directory(".")
      .changes(source)
  }
}
```

The `.changes(source)` call computes the diff between the generated output and the original source. `dagger generate` runs all generators and presents the unified changeset for review.

## Using other modules

Modules can depend on other modules. Add a dependency:

```shell
dagger module install github.com/dagger/go
```

Then call it in your code:

```dang
type MyCi {
  pub workspace: Directory! @defaultPath(path: "/")

  pub devContainer: Container! {
    # 'go' is the installed module — call it like a function
    go(source: workspace).env.withWorkdir("/app")
  }

  pub test: Void @check {
    devContainer
      .withExec(["go", "test", "./..."])
      .sync
    null
  }

  pub generate: Changeset @generate {
    devContainer
      .withExec(["go", "generate", "./..."])
      .withExec(["go", "mod", "tidy"])
      .directory(".")
      .changes(workspace)
  }
}
```

### Module references

A module reference follows the format `[proto://]host/repo[/subpath][@version]`:

```shell
github.com/shykes/daggerverse/hello@v0.3.0
#          ^^^^^^ ^^^^^^^^^^^^^ ^^^^^ ^^^^^^
#          host   repo          path  version
```

- `proto://` is optional (`ssh://` or `https://`). If omitted, Dagger chooses based on available authentication.
- `@version` can be a tag, branch, or commit. If omitted, the default branch is used.
- Private modules work with both HTTPS (Git credential managers) and SSH (`SSH_AUTH_SOCK`).

### Managing dependencies

```shell
# Install a dependency
dagger module install github.com/shykes/daggerverse/hello@v0.3.0

# Update to latest version
dagger module update hello

# Update to a specific version
dagger module update hello@v0.4.0

# Remove a dependency
dagger uninstall hello

# Local modules (same git repo) work too
dagger module install ./path/to/module
```

Dependencies are pinned to exact commits in `dagger.json` and versioned via git refs. Updates are explicit.

## Services

Start services for integration tests or dev environments:

```dang
type MyCi {
  let source: Directory! @defaultPath(path: "/")

  # Private service — not directly callable
  let db: Service {
    container
      .from("postgres:16")
      .withEnvVariable("POSTGRES_PASSWORD", "test")
      .withExposedPort(5432)
      .asService
  }

  pub integrationTest: Void @check {
    container
      .from("golang:1.22")
      .withDirectory("/app", source)
      .withServiceBinding("db", db)
      .withEnvVariable("DATABASE_URL", "postgres://postgres:test@db:5432/postgres")
      .withExec(["go", "test", "-tags=integration", "./..."])
      .sync
    null
  }
}
```

Services are content-addressed — the same definition always gets the same hostname. No port conflicts, no flaky test infrastructure.

## Secrets

Accept secrets as typed arguments — never as plain strings:

```dang
type MyCi {
  pub deploy(
    """
    API token for deployment
    """
    token: Secret!,
  ): Void {
    container
      .from("alpine")
      .withSecretVariable("DEPLOY_TOKEN", token)
      .withExec(["sh", "-c", "deploy --token=$DEPLOY_TOKEN"])
      .sync
    null
  }
}
```

The `Secret` type ensures tokens are never leaked to logs, caches, or filesystem layers. Dagger automatically scrubs secrets from all output streams, including crash reports.

Callers provide secrets via providers:

```shell
dagger call deploy --token=env:DEPLOY_TOKEN        # environment variable
dagger call deploy --token=file:./token.txt         # file
dagger call deploy --token=cmd:"gh auth token"      # command output
dagger call deploy --token=op://vault/item/field    # 1Password
dagger call deploy --token=vault://path/to/secret   # HashiCorp Vault
dagger call deploy --token=gcp://secret-name        # Google Cloud Secret Manager
```

**Security considerations:**
- Never write secret plaintext to files inside containers — it could be stored in the cache.
- Secrets are scoped to the module that defines them. To share across modules, pass them explicitly via function arguments.

Dagger has multiple layers of caching that work together to minimize redundant work.

### Layer caching

Every container operation (`from`, `withExec`, `withDirectory`, etc.) is automatically cached by its inputs. If the inputs haven't changed, the operation is skipped entirely. This is the same model as Docker layer caching, but applied to the entire Dagger API.

You don't need to do anything to enable layer caching — it's always on.

### Cache volumes

Use cache volumes for package manager caches and other persistent data that should survive across runs:

```dang
type MyCi {
  let source: Directory! @defaultPath(path: "/")

  pub build: Container! {
    container
      .from("node:20")
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withMountedCache("/app/node_modules", cacheVolume("node-modules"))
      .withExec(["npm", "install"])
      .withExec(["npm", "run", "build"])
  }
}
```

`cacheVolume("name")` is keyed by name — same name, same cache. Cache volumes are scoped to the module that defines them. To share a cache volume across modules, pass a reference to it via function arguments.

### Function caching

By default, Dagger caches function results for up to 7 days. If a function is called again with the same inputs (arguments, parent object state, module source), the cached result is returned without re-executing.

You can control this per function:

```dang
# Cache for 10 minutes (useful for external data that changes)
pub latestRelease: String! @cache(ttl: "10m") {
  # fetch from GitHub API...
}

# Cache only for the current session
pub sessionId: String! @cache(scope: "session") {
  # generate once per session...
}

# Never cache (always re-execute)
pub currentTime: String! @cache(scope: "never") {
  # always fresh...
}
```

**How it fits together:** A function cache hit skips the function entirely. A function cache miss runs the function, but individual operations inside it may still hit the layer cache. `@cache(scope: "never")` forces the function to run every call, but does *not* disable layer caching for the operations inside it.

## Configuration with defaults

Constructor arguments with defaults give users knobs to turn without requiring them:

```dang
type MyCi {
  pub source: Directory! @defaultPath(path: "/")
  pub nodeVersion: String! = "20"
  pub registry: String! = "ghcr.io"

  pub build: Container! {
    container
      .from("node:" + nodeVersion)
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["npm", "install"])
      .withExec(["npm", "run", "build"])
  }

  pub publish(tag: String!): String! {
    build
      .publish(registry + "/myorg/myapp:" + tag)
  }
}
```

Users can override defaults via the CLI or workspace config:

```shell
# CLI override
dagger call --node-version=18 build

# Or configure in .dagger/config.toml
# [modules.my-ci.config]
# node-version = "18"
# registry = "docker.io"
```

## Custom types

Modules can define custom types to model their domain. This is how you design the artifacts your module produces and consumes.

```dang
type MyCi {
  """
  Build result containing the binary and metadata
  """
  type BuildResult {
    pub binary: File!
    pub version: String!
    pub platform: String!
  }

  pub build(
    platform: String! = "linux/amd64",
  ): BuildResult! {
    let bin = container
      .from("golang:1.22")
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withExec(["go", "build", "-o", "/out/app", "."])
      .file("/out/app")

    BuildResult {
      binary: bin,
      version: "1.0.0",
      platform: platform,
    }
  }
}
```

Custom types are accessible through chaining, starting from a function on the main type. Dagger prefixes their names in the API schema (e.g. `MyCiBuildResult`) to prevent conflicts when multiple modules are loaded together.

Use custom types when:
- A function needs to return multiple related values
- You want to model a domain concept (a build result, a deployment target, a test report)
- You want to enable chaining on your return values

## Enumerations

Use enums to restrict string arguments to a set of valid values:

```dang
type Security {
  enum Severity {
    UNKNOWN
    LOW
    MEDIUM
    HIGH
    CRITICAL
  }

  pub scan(
    ref: String!,
    severity: Severity!,
  ): String! {
    container
      .from("aquasec/trivy:latest")
      .withExec(["trivy", "image", "--severity", severity, ref])
      .stdout
  }
}
```

Enum values are strictly validated. Passing an invalid value produces a clear error showing the allowed choices:

```shell
dagger call scan --ref=alpine:latest --severity=FOO
# Error: value should be one of UNKNOWN, LOW, MEDIUM, HIGH, CRITICAL
```

## Interfaces

Interfaces let your module accept arbitrary types from other modules without being tightly coupled to them:

```dang
type Deployer {
  """
  Any object that can produce a container image
  """
  interface Buildable {
    build: Container!
  }

  pub deploy(app: Buildable!, registry: String!): String! {
    app.build.publish(registry + "/app:latest")
  }
}
```

Any module that has a `build` function returning `Container!` can be passed as a `Buildable`. Dagger automatically detects compatible types and generates the necessary conversions — no explicit "implements" declaration needed by the other module.

Interfaces are useful for creating modules that compose with others without knowing about them in advance.

## Documenting your module

Descriptions on modules, functions, and arguments appear in `dagger functions`, `dagger call --help`, and the Daggerverse. Write them as part of your code:

```dang
"""
CI pipeline for my web application.
Handles building, testing, and publishing.
"""
pub description = "CI for my web app"

type WebCi {
  """
  Build the application for production
  """
  pub build(
    """
    Node.js version to use for the build
    """
    nodeVersion: String! = "20",
  ): Container! {
    # ...
  }
}
```

Good descriptions make your module self-documenting. Users can discover what it does and how to call it without reading source code.

## Putting it together

Here's a real-world module combining checks, generators, services, and configuration:

```dang
pub description = "CI for my web application"

type WebCi {
  let source: Directory! @defaultPath(path: "/") @ignorePatterns(patterns: [
    "node_modules", ".git", "dist", ".next",
  ])
  pub nodeVersion: String! = "20"

  # --- Shared infrastructure ---

  let baseImage: Container {
    container
      .from("node:" + nodeVersion)
      .withDirectory("/app", source)
      .withWorkdir("/app")
      .withMountedCache("/app/node_modules", cacheVolume("node-modules"))
      .withExec(["npm", "ci"])
  }

  let db: Service {
    container
      .from("postgres:16")
      .withEnvVariable("POSTGRES_PASSWORD", "test")
      .withExposedPort(5432)
      .asService
  }

  # --- Checks ---

  pub lint: Void @check {
    baseImage.withExec(["npm", "run", "lint"]).sync
    null
  }

  pub typecheck: Void @check {
    baseImage.withExec(["npx", "tsc", "--noEmit"]).sync
    null
  }

  pub unitTest: Void @check {
    baseImage.withExec(["npm", "test"]).sync
    null
  }

  pub integrationTest: Void @check {
    baseImage
      .withServiceBinding("db", db)
      .withEnvVariable("DATABASE_URL", "postgres://postgres:test@db:5432/test")
      .withExec(["npm", "run", "test:integration"])
      .sync
    null
  }

  # --- Generator ---

  pub generateApi: Changeset @generate {
    baseImage
      .withExec(["npm", "run", "generate:api"])
      .directory(".")
      .changes(source)
  }

  # --- Build & publish ---

  pub build: Container! {
    baseImage.withExec(["npm", "run", "build"])
  }

  pub publish(tag: String!, token: Secret!): String! {
    build
      .withRegistryAuth("ghcr.io", "dagger", token)
      .publish("ghcr.io/myorg/myapp:" + tag)
  }
}
```

```shell
# Run all checks
dagger check

# Run generators
dagger generate

# Build
dagger call build

# Publish
dagger call publish --tag=v1.0.0 --token=env:GITHUB_TOKEN
```

## Testing

Test your module by calling its functions:

```shell
# Smoke test — does it build?
dagger call build

# Run checks
dagger check

# Run generators and verify output
dagger generate
git diff --exit-code
```

For more thorough testing, write a separate test module that imports yours and validates behavior.

## Dang vs. other SDKs

| | Dang | Go | Python | TypeScript |
|---|---|---|---|---|
| Codegen required | No | Yes | Yes | Yes |
| External libraries | No | Yes | Yes | Yes |
| Runtime overhead | None | Minimal | Moderate | Moderate |
| File count | 1 (`main.dang`) | 3+ | 3+ | 3+ |
| Learning curve | Dagger concepts only | Dagger + Go | Dagger + Python | Dagger + TS |
