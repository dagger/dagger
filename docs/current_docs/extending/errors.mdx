# Errors

Errors are part of the module API. A great module does not just fail accurately; it helps the caller fix the problem.

## Error Design Goals

A module error should answer:

- What failed?
- Which input, file, service, secret, or external system caused it?
- Is the problem in the caller's workspace, configuration, credentials, network, or the module?
- What should the caller try next?
- Where can they look for more detail?

## Common Error Categories

Most module errors fall into a small set of categories. Naming the category helps authors write better messages:

| Category | What the caller needs |
| --- | --- |
| Invalid input | Which argument or setting is wrong, the accepted shape, and an example value if useful. |
| Missing workspace content | Which file or directory was expected, from which workspace path, and whether a default path was used. |
| Missing or invalid secret | Which credential is needed, what scope it needs, and how to pass it without exposing the value. |
| Service readiness | Which service failed to start, which port or health condition was expected, and where startup logs are visible. |
| External system failure | Which system rejected the request and whether retrying, changing credentials, or changing configuration is likely to help. |
| Module bug | Enough context for maintainers to reproduce, without blaming the caller for an internal failure. |

## Prefer Caller Language

Do not surface internal implementation names when the caller needs a product-level explanation.

For example, a deployment module should not fail with only a low-level HTTP or CLI error if it can explain that the registry token is missing, expired, or scoped incorrectly.

## Make Validation Early and Specific

Validate inputs close to the module boundary when possible.

Good validation:

- Names the invalid input.
- States the accepted shape.
- Does not start expensive work first.
- Avoids leaking secrets.
- Preserves useful lower-level context.

## Use Traces as Teaching Tools

Dagger traces are part of the diagnostic experience. Module authors should structure work so traces tell a coherent story:

- Use meaningful function and object names.
- Group steps in a way that matches the caller's mental model.
- Avoid burying the real failure under incidental setup.
- Return structured diagnostic outputs when useful.
