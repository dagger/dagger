# Generated Code Lifecycle

Most SDK-backed modules include code generated by Dagger. Generated code gives the module type-safe access to the Dagger API, installed dependencies, and custom types from other modules.

## When to regenerate

Run a module-loading command such as `dagger functions -m .` after changes that affect the module schema or SDK view:

- adding, removing, or renaming public functions
- changing function arguments or return types
- adding, removing, or updating module dependencies
- changing the module SDK or SDK configuration
- pulling a Dagger version that changes generated API types

Run `dagger functions -m .` before opening a large module change in your IDE if completions look stale.

## What to commit

Commit source files, module configuration, examples, and tests.

Whether generated SDK files are committed depends on the SDK and generated path:

| SDK | Generated path | Commit expectation |
| --- | --- | --- |
| Dang | None | No generated SDK files. |
| Go | SDK-managed files in the module directory | Follow the generated `.gitignore`; commit files that are not ignored. |
| TypeScript | SDK package and path mappings under the module | Follow the generated `.gitignore`; commit source and package metadata, not ignored dependency directories. |
| Python | local SDK package under `sdk` | Follow the generated `.gitignore`; install it for IDE support. |
| Java | `target/generated-sources` | Do not commit; `target` is ignored by default. |
| PHP | local package under `sdk` | Follow the generated Composer path repository setup. |

If the generated files are ignored by the module template, do not force-add them. If they are not ignored, treat them as part of the module's checked-in API surface and review the diff.

## Keeping generated code current

Use this loop while authoring:

```shell
dagger functions -m .
dagger call <function>
dagger check
```

For CI, add a job that regenerates and verifies the working tree stays clean:

```shell
dagger functions -m .
git diff --exit-code
```

This catches forgotten generated-code updates before users install the module.

## Generated assets

Generated SDK files are different from generated project assets. Project assets are produced by module functions annotated as generators and run through:

```shell
dagger generate
```

Use `dagger generate` for files users should review and apply as changesets, such as API clients or scaffolding. Use a module-loading command such as `dagger functions -m .` for SDK support files that let the module compile and provide IDE completions.
