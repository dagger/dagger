# Module Developer Guide

This guide is for developers who want to create Dagger modules.

A module is a reusable API for automation. It can expose functions, objects, checks, services, generators, and typed workflows that other people can run from a workspace.

This guide does not teach one SDK. It teaches the Dagger model every SDK shares.

## Start Here

Read the guide in this order:

1. **Learn how Dagger works.** Workspaces, modules, functions, checks, cache, and execution.
2. **Decide what to build.** Know when a workflow deserves a module.
3. **Shape the API.** Choose the right functions, objects, inputs, outputs, and defaults.
4. **Use the type system.** Make the module easy to call and compose.
5. **Design for real use.** Errors, checks, services, secrets, Cloud, and quality.
6. **Choose an SDK.** Use SDK docs for syntax, file layout, package tools, and tests.

## What This Guide Does Not Do

This guide avoids universal pseudocode. Code examples belong in SDK docs, where syntax, package layout, generated clients, and tests can be taught honestly.

It also avoids command-driven teaching. Commands help you get started. A module author needs the deeper model: how workspaces resolve modules, how functions build the graph, when work runs, why cache works, and how checks reach CI and Cloud.

## SDK-Specific Documentation

SDK docs should stand alone. Their reader has learned the platform model here and now needs one language.

Use the [SDK documentation structure](./sdks/index.mdx) as the editorial blueprint for those standalone docs.
