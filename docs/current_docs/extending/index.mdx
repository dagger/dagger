# Developing Modules

When pre-built modules don't cover your needs, you can write your own. A module is a package of functions — written in any supported SDK — that extends the Dagger API.

## When to develop a module

**Install first.** Browse the [Daggerverse](https://daggerverse.dev) and check if someone has already built what you need. `dagger install` is always faster than writing code.

**Then customize.** If an existing module is close but not quite right, you can often configure it via your workspace config rather than forking it.

**Then build.** Write a module when:

- Your workflow is specific to your project and no general module fits
- You need to compose multiple tools in a way that's unique to your stack
- You want to enforce team conventions (naming, structure, policies)
- You want to share reusable automation with the community

## The progression

Most modules start small and grow:

1. **Project module** — Lives in your repo, serves your team. Quick to write, doesn't need to be general.
2. **Team module** — Shared across repos in your org. More polished API, configuration options.
3. **Community module** — Published for anyone to use. Well-documented, versioned, tested.

You don't have to aim for step 3. A project module that makes your team faster is valuable on its own.

## Choose an SDK

| SDK | Best for | Trade-off |
|-----|----------|-----------|
| **Dang** | Dagger-native workflows, fastest path | DSL — no external libraries |
| **Go** | Performance, systems work, existing Go codebases | Verbose, requires codegen |
| **Python** | Scripting, data workflows, existing Python tools | Runtime overhead |
| **TypeScript** | Web toolchains, existing Node.js tools | Runtime overhead |

**Dang** is Dagger's own DSL. It maps directly to the Dagger API with no codegen, making it the fastest way to write a module. If your module is primarily orchestrating containers and files, Dang is the right choice.

If you need to call external libraries (e.g., a Go parser, a Python ML library), use the corresponding SDK.

## Get started

```shell
# Create a new module
dagger module init --sdk=go my-module

# Or with Dang
dagger module init --sdk=dang my-module
```

This creates a module in `.dagger/modules/my-module/` with scaffolding for your chosen SDK.
