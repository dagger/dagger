# Module Developer Guide

This guide is for developers who want to create Dagger modules: reusable packages of functions, types, checks, services, and generators that other people can run from a workspace.

The goal is not to teach one SDK. The goal is to teach the Dagger platform model that every SDK exposes: what a workspace is, what a module is, how functions become a typed API, how the engine evaluates a graph, how checks become workspace automation, and how to design modules that are easy to call, test, document, and compose.

This guide now includes the platform fundamentals that used to live in a separate Core Concepts section. That material belongs here because module development is not separable from the way Dagger works under the hood. If you do not understand workspaces, module loading, function calls, checks, cache, and materialization, you cannot reliably design a module API. If you do understand those mechanics, module design decisions become much easier to explain.

## How to Use This Guide

Read this guide before choosing SDK-specific documentation. A module developer should finish it able to answer these questions:

- How does a workspace install, name, configure, check, and automate a module?
- What problem should this module solve, and who is it for?
- What should be a function, an object, a setting, a check, a service, or a generator?
- What inputs should be explicit, and what can safely come from workspace defaults?
- How do Dagger objects, directories, files, containers, secrets, services, and changesets behave?
- How do modules compose with other modules?
- What makes a module predictable, cache-friendly, debuggable, and pleasant to call?

## What This Guide Does Not Do

This guide intentionally avoids using Dang as universal pseudocode. Code examples belong in SDK documentation, where the syntax, package manager, project layout, generated files, and testing tools can be taught honestly.

It also does not teach module development as a pile of command snippets. CLI commands are useful for onboarding, but module authors need a durable mental model first: how workspaces resolve modules, how functions build the graph, when work materializes, why cache keys change, and how checks travel from local development into Cloud automation.
