# Developing Modules

A module is a package of Dagger functions. To write one, start by choosing an SDK.

## Author reusable modules

- [Building Reusable Modules](/extending/reusable-modules): Design, test, version, and release modules for other workspaces.
- [Generated Code Lifecycle](/extending/generated-code): Know when to refresh SDK-managed files and what generated files to commit.
- [SDK Status](/extending/sdk-status): Compare edition maturity, generated-code behavior, and best fit.
- [Testing Modules](/extending/testing): Test reusable modules and examples before release.

## Choose an SDK

- [Base Edition](/extending/editions/dang): Dang. Smallest setup; best for pure Dagger orchestration.
- [Go Edition](/extending/editions/go): Go. Strong types, compiled tools, Go libraries.
- [TypeScript Edition](/extending/editions/typescript): TypeScript. Node, Bun, Deno, frontend tooling.
- [Python Edition](/extending/editions/python): Python. Python packages, data tooling, scripts.
- [Java Edition](/extending/editions/java): Java. IDE setup is documented; the full guide is not yet available.
- [PHP Edition](/extending/editions/php): PHP. IDE setup is documented; the full guide is not yet available.

Not sure? Start with Dang. Use another SDK when you need that language's ecosystem.

## What SDK Pages Cover

Each SDK page owns its own setup:

- installing the SDK
- initializing a new module
- regenerating SDK-managed files
- adding dependencies
- IDE setup and project layout
