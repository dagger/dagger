# Interfaces and Composition

Modules become more valuable when they can compose. Interfaces let module authors define contracts that multiple modules can implement and consume.

Use interfaces when the caller cares about capability, not implementation.

## Why Interfaces Matter

Interfaces help with:

- Swapping implementations.
- Testing with smaller fakes or fixtures.
- Supporting several tools behind one workflow.
- Building higher-level modules that accept lower-level capabilities.
- Avoiding hard dependencies on one module.

## Interface Design Principles

Good interfaces are small. They describe one capability in caller language.

Prefer interfaces that:

- Have a narrow purpose.
- Return Dagger objects or structured results.
- Avoid leaking implementation-specific settings.
- Can be implemented by more than one module.
- Can be tested independently.

Avoid interfaces that:

- Become a dumping ground for every method in a module.
- Mirror one SDK's class shape.
- Require callers to know which implementation is underneath.

## When Not to Use an Interface

Do not introduce an interface just because two modules have similar names. An interface is worth the extra abstraction when the caller benefits from substituting implementations.

Keep a concrete dependency when:

- There is only one real implementation.
- The consumer needs implementation-specific behavior.
- The interface would only rename existing functions.
- The module API is still changing quickly.
- The abstraction would make traces or errors harder to understand.
