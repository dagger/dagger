# Module Anatomy

A module has two audiences:

- Dagger, which needs metadata and source code it can load.
- People, who need a stable API, clear documentation, and predictable behavior.

The physical files matter, but they are not the point. The point is the API that the module exposes.

## Required Pieces

Every module has:

- A `dagger.json` file.
- A module name.
- An SDK declaration.
- Source files for the selected SDK.

Most useful modules also have:

- Dependencies on other modules.
- Documentation for the module, functions, arguments, and return values.
- Checks or tests.
- Examples.
- A release or versioning strategy.

## Public API and Private Implementation

The public API is what Dagger exposes to callers: module objects, functions, arguments, return types, docs, checks, generators, and settings.

The private implementation is everything callers should not have to care about: helper functions, intermediate containers, package manager commands, generated SDK support files, fixtures, and internal dependency layout.

Good modules keep that boundary clean. If a caller needs to know an internal path, helper name, temporary container, or generated-file detail to use the module, the API is probably leaking implementation.

## What Belongs in `dagger.json`

`dagger.json` is module metadata. It belongs to the module, not the workspace.

Use it for:

- Module name.
- SDK source.
- Module source directory.
- Included and ignored files for module loading.
- Module code dependencies.
- Engine compatibility.

Do not use it for workspace installation, workspace settings, environments, or team-level project configuration. Those belong in `.dagger/config.toml`.

## API Lifecycle

A module API has its own lifecycle:

- **Sketch**: discover the user workflow and function names.
- **Prototype**: expose the smallest useful function.
- **Harden**: narrow inputs, improve errors, add settings, add tests.
- **Document**: write function and argument docs that appear in CLI help.
- **Release**: pin dependencies, tag versions, communicate breaking changes.
- **Operate**: observe traces, maintain compatibility, keep examples current.
