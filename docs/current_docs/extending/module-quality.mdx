# Quality

A module is successful when someone else can call it, trust it, debug it, and upgrade it.

Quality is not a final polish step. It is part of module design.

## Definition of Done

A module is ready for other people when it has:

- A small public API with names that match caller intent.
- Function and argument documentation that makes CLI help useful.
- Typed inputs and outputs that compose with other Dagger calls.
- Explicit secrets, services, files, directories, settings, and external side effects.
- Tests for the main path and the most important failure modes.
- Checks or generators when the module is meant to automate workspace health or source changes.
- Errors that tell callers what to fix.
- Trace output that tells a coherent story.
- Examples that use the current command surface.
- Versioning and release notes for caller-visible changes.

## Designing for Testability

Practical test harnesses are SDK-specific. Test files, assertions, generated clients, async behavior, package managers, and fixtures belong in the SDK docs.

At the platform level, testability starts with module design:

- Keep the public API small enough to exercise directly.
- Put important behavior behind typed functions, not hidden shell scripts.
- Make files, directories, services, secrets, settings, and external systems explicit.
- Return structured values that tests can inspect.
- Validate inputs early so failure cases are cheap to test.
- Keep side effects in clearly named functions.
- Use checks for project health signals that should run in automation.

## What Deserves Coverage

When writing SDK-specific tests, module authors should cover:

- Function behavior.
- Default settings.
- Important failure modes.
- Composition with dependencies.
- Generated changesets.
- Services and readiness.
- Secrets without exposing secret values.

## Documentation

Documentation lives in several places:

- Module overview.
- Function docs.
- Argument docs.
- Return value docs.
- Examples.
- Release notes.
- Troubleshooting notes.

The CLI is a documentation surface. `dagger functions` and `dagger call --help` should be useful without sending the reader to a website for every question.

## Examples

Examples should be small, real, and maintained. A great example teaches one concept and gives the reader a path to adapt it.

Avoid examples that require a large repo, an unstated credential, or an old command surface.

## Compatibility

Treat the module API as a contract. Breaking changes should be deliberate.

Before release, ask:

- Did a function or argument name change?
- Did a return type change?
- Did default behavior change?
- Did a setting move?
- Did a dependency update alter caller-visible behavior?
- Are examples and checks still accurate?
