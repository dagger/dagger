# Objects, State, and Persistence

Dagger objects let a module represent a workflow concept and expose methods on it. This is powerful, but it should be used deliberately.

A Dagger object is not a mutable object in the usual application sense. Module APIs should be designed around values that are clear to the caller and reproducible to the engine.

## What Belongs in an Object

Use an object when several functions share a meaningful context:

- A configured toolchain.
- A release with a version, source directory, and target registry.
- A deployment environment.
- A build matrix.
- A service stack.
- A test report.

The object should answer: "What concept am I holding on to?"

## Constructor State

Constructor arguments are a good place for durable context:

- Source directory.
- Tool version.
- Target platform.
- Registry.
- Environment name.
- Feature toggles.

Constructor state should be visible and explainable. If the caller cannot tell why changing it changes later results, it probably should not be constructor state.

## State Design Checklist

Before adding state to an object, ask:

- Is this state part of the caller's mental model?
- Will changing it change cache keys or outputs in a way the caller can predict?
- Should the state be an explicit constructor argument, a workspace setting, or a function argument?
- Does the object still make sense when passed to another module?
- Can the state be represented by Dagger values such as `Directory`, `Container`, `Secret`, or `Service`?
- Would the trace explain where the state came from?

## Avoid Hidden Mutation

Do not design APIs where methods appear to mutate invisible state and callers have to guess what happened. Prefer methods that return the next object or a concrete output.

Good Dagger APIs make the chain visible:

- Configure.
- Build.
- Test.
- Publish.
- Export.
