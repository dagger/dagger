## About Dang

Dang is Dagger's native DSL. It maps directly to the Dagger API with no codegen — what you write is what runs. There's no build step, no generated files, no runtime overhead.

## Project setup

```shell
dagger module init --sdk=dang my-module
```

This creates a module at `.dagger/modules/my-module/` with a single `main.dang` file.

## Language basics

Dang is intentionally small. The entire language:

- **Types** are declared with `type Name { ... }`
- **Public members** use `pub`, private use `let`
- **Functions** are type members that return a value: `pub build: Container! { ... }`
- **Arguments** go in parentheses: `pub build(source: Directory!): Container! { ... }`
- **Annotations** modify behavior: `@check`, `@generate`, `@defaultPath`, `@ignorePatterns`
- **Non-null** is marked with `!`, nullable is the default
- **Descriptions** use triple-quoted strings: `""" ... """`
- **Comments** use `#`
- **Module metadata** uses top-level `pub`: `pub description = "..."`

## When to use Dang

Use Dang when your module is primarily orchestrating containers, files, and other Dagger objects. Dang has no package manager and no external library ecosystem — if you need to call a Go parser, a Python ML library, or a Node.js bundler API directly, use the corresponding SDK instead.

## Dang vs other SDKs

| | Dang | Go | Python | TypeScript |
|---|---|---|---|---|
| Codegen required | No | Yes | Yes | Yes |
| External libraries | No | Yes | Yes | Yes |
| Runtime overhead | None | Minimal | Moderate | Moderate |
| File count | 1 (`main.dang`) | 3+ | 3+ | 3+ |
| Learning curve | Dagger concepts only | Dagger + Go | Dagger + Python | Dagger + TS |
