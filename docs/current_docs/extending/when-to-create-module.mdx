# When to Create a Module

A module is the right tool when a workflow should become a stable interface for people, agents, or automation to call repeatedly.

Start with the user. A good module saves someone from remembering a sequence of steps, environment assumptions, credentials, files, tools, or conventions. It gives that person a small API that does the right thing consistently.

## Good Reasons to Create a Module

Create a module when the work has one or more of these properties:

- It is run often enough that a named function is better than a remembered command.
- It has project conventions that should be encoded once and reused.
- It needs Dagger features such as containers, secrets, services, directories, checks, generators, or changesets.
- It will be called in more than one context: locally, in CI, in Cloud checks, or by an agent.
- It needs to compose with other modules.
- It should be discoverable through `dagger functions`, `dagger call --help`, `dagger check`, or `dagger generate`.
- It should hide infrastructure details behind a product-specific workflow.

## Poor Reasons to Create a Module

Do not create a module just to wrap one obvious command with no durable semantics. A thin wrapper can still be useful, but it should earn its name by improving the caller's experience.

Avoid modules that:

- Mirror every internal shell script without improving names, inputs, outputs, or errors.
- Require callers to know internal file layout that the module could discover or default.
- Depend on unspoken host state.
- Use secrets or network access implicitly.
- Return logs where they should return structured values.
- Expose low-level implementation steps as the primary user interface.

## Module or Workspace Module?

A module is a package of code with a `dagger.json`.

A workspace module is a module installed into a workspace through `.dagger/config.toml`. Workspace modules are what project contributors call by name:

- `dagger call lint`
- `dagger call go test`
- `dagger check`
- `dagger generate`

| Need | Usually choose | Why |
| --- | --- | --- |
| Project contributors need a stable workflow for one repository. | Workspace module | The module becomes part of the workspace's normal command surface. |
| Several repositories or teams need the same capability. | Reusable module | The module can be versioned, installed, and composed from many workspaces. |
| You are experimenting with an external module or debugging a one-off call. | Direct module source call | The module does not yet need to become part of the workspace API. |
| You only need to run a simple local command once. | No module yet | A module should add a durable interface, not just ceremony. |

## A Useful First Design Question

Ask: "What would I want a new teammate to type without explanation?"

If the answer is a clear command, the module API probably starts there. Work backward from that call:

- What is the function name?
- What inputs does it need?
- Which inputs deserve defaults?
- Which outputs should be structured objects?
- Which failures should teach the caller what to do next?
- Which details should stay private inside the module?
