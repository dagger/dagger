# Dagger

:::tip Migrating from a previous version?
Dagger now uses **workspaces** (`.dagger/config.toml`) instead of configuring everything in `dagger.json`. Your existing setup still works, but see [Upgrading to Workspaces](/reference/upgrade-to-workspaces) when you're ready to migrate.
:::

Dagger is a CI orchestration engine. Define your pipelines once, in real code. Run them identically before and after push, locally or at scale.

## Why Dagger?

**Programmable.** CI has been stuck on shell scripts and proprietary YAML for too long. Dagger pipelines are real code: Go, Python, TypeScript, or 5 other languages. Types, tests, abstractions, and your editor — not a textarea in a browser.

**Local-first.** CI shouldn't start at `git push`. Run your full pipeline locally — same containers, same result — before you commit. Your laptop, an AI sandbox, a CI server, or dedicated cloud infrastructure. One engine, everywhere.

**Repeatable.** Every pipeline runs in containers. Dependencies are explicit and strictly typed. Caching is automatic with fine-grained control. A test result from your laptop and a test result from CI should be the same result. With Dagger, they are.

**Observable.** CI logs are a wall of text. You shouldn't have to guess what broke. Dagger gives you step-level tracing across your entire pipeline. See what happened at every step, then drill down to the one that failed.
