# Artifacts

An artifact is a thing in your workspace that Dagger can operate on as a unit — something you might check, ship, or generate code for. When you run `dagger check`, Dagger finds the artifacts in your workspace and runs their checks.

## Where artifacts come from

Each [module](/core-concepts/modules) you install contributes an artifact to your workspace. For example, installing the `eslint` module contributes an ESLint artifact; installing `vitest` contributes a Vitest artifact.

```shell
dagger install github.com/dagger/eslint
dagger install github.com/dagger/vitest
```

Your workspace now has two artifacts: one from each module.

## What artifacts do

Artifacts are the units that [verbs](/core-concepts/functions) operate on. A module author attaches functions to an artifact — a check function, a ship function, a generate function — and those functions run when you invoke the corresponding verb:

```shell
dagger check        # runs every check attached to every artifact
dagger check eslint # runs checks attached to the eslint artifact only
```

The artifact is how Dagger answers "what can I check here?" It's the selection model for lifecycle commands.

## Why this matters

Without artifacts, every verb command (`check`, `ship`, etc.) would need its own discovery logic and its own selection syntax. With artifacts, the selection vocabulary is shared: the same way you filter what to check, you filter what to ship.

This is also why installing a module feels like adding a unit of functionality rather than a bag of functions — the artifact is the unit.
