Design proposal

Engine-managed host volumes

Expose an operator-controlled directory beneath the Dagger engine state root as an opaque Volume, then recursively bind-mount it into containers. The feature is deliberately small: Dagger provides naming, safe containment, API plumbing, and mount lifecycle; the engine operator owns the backing storage and its lifecycle.

1. Executive summary and motivating scenarios

An engine operator may already attach storage to the engine container or host namespace. Dagger currently has no direct way to make that live, operator-managed directory available through the opaque Volume API. This proposal adds Query.engineVolume(name, subdir?). A named volume resolves beneath the configured engine root:

<engine-root>/volumes/v1/<name components>/fs[/<subdir>]

For the default root, engineVolume("datasets/models", subdir: "llama") mounts /var/lib/dagger/volumes/v1/datasets/models/fs/llama. The engine lazily creates the named volume root through fs with mode 0755; an explicitly selected subdirectory must already exist.

Concrete scenarios

  • Large local dataset. An operator bind-mounts a multi-terabyte disk at .../volumes/v1/datasets/fs. Workflows consume it without copying or snapshotting the tree.
  • Grouped operator configuration. An operator bind-mounts one parent directory at .../volumes/v1/team-a, supplying many volumes such as team-a/models and team-a/cache without configuring a host mount per volume. That parent contains the payload boundaries, for example models/fs and cache/fs.
  • Shared mutable tool state. Several containers mount the same named directory read-write and coordinate, if necessary, using the application’s own mechanisms. Dagger adds no locking.
  • Portable module. A main client constructs the volume and passes the opaque value to a module. The module can mount it but cannot construct an engine volume itself.

3. Goals and explicit non-goals

Goals Agreed

  • Return the existing opaque Volume type from a constructor specialized to engine-side storage.
  • Expose a live recursive bind of a directory controlled by the engine operator.
  • Support hierarchical names with a conservative portable grammar and a reserved fs payload boundary.
  • Keep the source contained beneath <engine-root>/volumes/v1, including during symlink resolution.
  • Allow main clients to construct; allow modules to accept and mount; prevent module construction as a product boundary.
  • Keep cache identity simple and stable: constructor arguments, not current contents.
  • Support both read-write and the existing withMountedVolume(readOnly: true) behavior.

Non-goals Agreed

  • Arbitrary paths supplied by a client; per-volume ACLs; module sandboxing; generic multi-tenant hardening.
  • Content hashing, filesystem watching, mutation-driven cache invalidation, snapshotting, or copying.
  • Engine-managed pruning, quotas, accounting, migration, backup, ownership changes, or cleanup of volume contents.
  • Dagger-provided locking or consistency for concurrent readers/writers.
  • Configurable propagation flags, overlay composition, metadata files, or automatic creation of subdir in v1.
  • Windows support in the initial Linux engine implementation.

4. User-facing API, CLI, and module behavior

GraphQL/API

engineVolume(name: String!, subdir: String): Volume!

The constructor is top-level and main-client-only. The returned value reveals no backend-specific fields; it is passed to the existing Container.withMountedVolume(path, volume, readOnly, expand).

// Main client
vol := dag.EngineVolume("datasets/models",
    dagger.EngineVolumeOpts{Subdir: "llama"})
out, err := dag.Container().From("alpine").
    WithMountedVolume("/models", vol,
        dagger.ContainerWithMountedVolumeOpts{ReadOnly: true}).
    WithExec([]string{"find", "/models", "-maxdepth", "1"}).Stdout(ctx)

Address/CLI form Recommended

dagger call consume --data 'engine-volume://datasets/models?subdir=llama'

Address.volume parses the address for CLI custom Volume arguments, then calls the same constructor semantics. Unknown parameters are errors. Rejecting repeated scalar parameters is a recommendation—stricter than the current SSHFS parser—not an agreed requirement. URL host/path ambiguity must not produce two spellings for the same name; the implementation should define and test one canonical parser. The exact scheme spelling remains an open naming choice.

Module behavior

Module introspection scrubbing must add Query.engineVolume; Address.volume is already hidden from module introspection as a whole. The fields still exist in the served DagQL schema, so runtime resolvers also call RequireMainClient. Generated module SDKs omit them. Modules still receive Volume parameters and call withMountedVolume. This mirrors SSHFS’s product boundary and is explicitly not an authorization boundary.

5. Terminology and precise bind semantics

Engine root
The configured BuildKit/Dagger root, defaulting to /var/lib/dagger; current server startup makes it absolute and resolves its symlinks.
Volume namespace root
<engine-root>/volumes/v1.
Name
One or more slash-separated components. Each component uses a conservative portable grammar (recommended: ASCII alphanumeric first, then ASCII alphanumeric, dot, underscore, or hyphen); empty, ., .., and reserved fs components are rejected.
Volume root
<namespace-root>/<name>/fs, lazily created if absent with 0755.
Selected root
The volume root plus an optional relative subdir. The subdirectory is payload navigation, not part of the volume name, and is never auto-created.

At exec preparation, the selected root becomes a containerd mount with bind type and rbind; ro is appended for a read-only container mount. Recursive bind makes operator-provided nested mount points visible. On current Linux/runc semantics, ro applies to the top-level bind but is not recursively imposed on submounts brought in by rbind; those retain their operator-configured read/write state. The recommended v1 behavior is to document that limit rather than require kernel-dependent recursive-read-only machinery. The engine adds no propagation mode in v1. The bind exists only for the exec/container lifetime; the backing directory persists.

Engine-created roots are normally root:root 0755. A non-root container user can read them but cannot write until the operator arranges suitable ownership or permissions; Dagger does not infer or rewrite ownership.

6. Real trust and security boundary

The engine operator chooses what exists or is mounted beneath volumes/v1. That operator already controls the engine process, its state volume, privileges, and namespace. Dagger intentionally trusts this setup. A main client may select only a validated name and subdirectory within that operator-curated tree; it may not submit an arbitrary host path. A module cannot construct the capability but may use one passed by its caller.

With a remote engine, “host” means the engine host/container namespace—not the CLI machine. An operator who exposes sensitive material beneath the namespace makes it selectable by main clients. This feature therefore belongs only on engines whose main clients are trusted for that capability. The module restriction improves product portability and intent; it is not an ACL and must not be described as one.

Path validation prevents traversal and uses the existing continuity safe-root semantics: symlinks are resolved as if chrooted within the containment root, so absolute targets and .. cannot escape it. An apparent outside target is clamped to an in-root path and may then fail as missing or the wrong type; it is not promised to produce a distinct “escape” error. Operators must use bind mounts—not symlinks—to redirect a volume hierarchy to storage outside volumes/v1. Symlinks inside the mounted payload retain ordinary filesystem behavior after the source has been safely selected. Because the operator is trusted, v1 does not promise protection from an operator concurrently replacing path components between validation and mount; claiming adversarial multi-tenant TOCTOU safety would require descriptor/openat2-based mounting and a broader threat model.

DagQL cache hits do not necessarily rerun a resolver. With stable identity, a hand-crafted module query matching a previously cached constructor call could receive that cached value without executing RequireMainClient. That is acceptable here precisely because construction hiding is a product boundary, not an authorization boundary.

7. End-to-end architecture and flow

Main client / CLI
DagQL constructor
Opaque Volume payload
Exec mount preparation
BuildKit executor / rbind
  1. The SDK or Address.volume sends canonical name and optional subdir. No caller-host filesync or host service is involved.
  2. The schema resolver enforces main-client use, validates canonical path syntax, and returns a Volume that implements persisted-object encoding so it can participate in persistence closures. Its backend payload contains name and subdirectory—not an already-resolved live host path. Recording layout version in that payload is recommended but was not part of the explicit agreement.
  3. withMountedVolume stores the opaque volume as an owned container dependency, exactly as for other volume backends.
  4. At exec time, prepareExecVolumeMount dispatches to the engine-volume backend. It derives the namespace from the server’s configured root, creates the named root if needed, safely resolves the selected source, verifies it is a directory, and returns a recursive bind mount.
  5. BuildKit’s gateway/executor mounts it into the exec namespace. Existing exec teardown unmounts the bind. There is no helper process, service, secret material, FUSE mount, or long-lived per-session resource to stop.

Lifecycle and cancellation

Construction is declarative and performs no mount. Cancellation before exec leaves at most the lazily created empty volume directory. Cancellation, client disconnect, exec failure, or engine shutdown after mounting follows the existing executor mount teardown; unmounting does not remove backing data. A later session can reuse the same recipe/name. Mount cleanup must be idempotent, but there is no backend-specific goroutine or process cleanup.

The constructor belongs to the engine/server surface, not the caller-host Host service: session/workspace services identify the caller machine, whereas this source is deliberately tied to the engine installation. RequireMainClient denies every same-session nested client, whether a module runtime or a nested CLI. A client that opens its own separate session is that session’s main client and may construct the volume.

AreaInvariant

ContainmentName stays beneath volumes/v1; subdir stays beneath that volume’s fs. Absolute/traversal syntax fails; symlinks resolve chroot-style within the appropriate containment root and cannot select an outside source.

LifetimeBind is prepared per exec and released with it. Backing storage is operator-owned and survives cancellation, disconnect, shutdown, and restart.
CacheNo content read/hash/watch. Identical exec recipes may return a cached result despite backing mutations; cache-busted/new execs observe current contents. This matches agreed SSHFS semantics.

PlatformThe engine is Linux-only today; v1 relies on Linux bind-mount semantics and defines no copy fallback.

9. Relationship to merged SSHFS volumes

The merged SSHFS work established the opaque Volume, persisted backend configuration, Container.withMountedVolume, backend dispatch at exec time, Address.volume, module-constructor hiding plus runtime gating, generated SDK surfaces, and authentic integration patterns. Those primitives should be extended, not duplicated.

SSHFSEngine volume
SourceRemote SSH/SFTP endpointOperator-curated engine filesystem
DependenciesSecrets, optional service; session-scoped by defaultNo session resource; stable constructor identity
SetupTmpfs credentials, sshfs process, FUSE mountSafe path resolution and recursive bind
CleanupUnmount FUSE, stop process/service, scrub tmpfsExecutor unmount only; never delete backing
Failure surfaceNetwork/auth/process/host-key failuresPath/type/permission/mount failures

It is therefore not “SSHFS with another address.” Reuse the volume/API/mount abstraction, while keeping the backend’s ownership and lifecycle minimal.

10. Compatibility, version gating, and schema

  • All new public GraphQL fields and arguments are gated with AfterVersion("v1.0.0-0"), following current Dagger policy. Module introspection filters must omit the constructor.
  • The persisted backend payload includes an explicit layout version/kind. It stores logical name/subdir rather than an absolute root, so custom engine roots and restarts remain correct.
  • On-disk v1 allows future layouts to coexist; it does not commit Dagger to a speculative migration. A future engine may continue resolving v1 and separately introduce v2.
  • The existing Volume API remains opaque and backend-neutral. No SSHFS-specific specialization leaks into the return type.
  • SDKs, schema docs, CLI custom argument parsing, and module SDK filtering are regenerated through dagger generate.

12. Performance grounded in expected workloads

The motivating workloads are large trees where importing or hashing is the cost to avoid. Construction and mount preparation perform bounded work proportional to path depth: validation, a small number of directory operations, safe resolution, stat, and one recursive bind. They do not walk payload contents. Data I/O follows the backing filesystem directly.

Large files and trees therefore do not add control-plane work. Concurrent mounts consume ordinary kernel mount entries and share backing bandwidth. Backpressure, coherence, and filesystem performance are supplied by the backing filesystem and applications. No speculative index, watcher, pool, or content cache belongs in v1.

13. Detailed implementation map and coarse chunks

Chunk 1 — complete backend behavior

Meaningful increment: a main-client GraphQL call can construct, persist, mount, mutate, remount, and read an engine volume with correct containment and lifecycle.

  • core/volume.go: add an engine backend kind/config (canonical name, optional subdir, and recommended explicit layout version), persisted encoding/decoding, dependency behavior, and stable identity. Do not add session dependencies.
  • core/schema/volume.go: add version-gated engineVolume, validation, docs, and RequireMainClient. Prefer a shared validation helper with table tests. Add Query.engineVolume to module-introspection scrubbing in core/moddeps.go in this same chunk.
  • core/query.go and engine/server: expose the configured engine state root through the narrowest core server interface needed; do not hard-code /var/lib/dagger.
  • core/volume_mount.go: backend dispatch; lazy 0755 creation; safe-root resolution using the established containerd continuity helper pattern; directory checks; rbind plus optional ro. Return an ordinary mountable whose unmount is owned by the executor.
  • Focused core/volume*_test.go and core/schema/volume_test.go: persistence, dependency absence, canonical grammar, reserved fs, traversal/symlink containment, creation races, existing-mode preservation, mount options, and main-client gate.

Exit criteria: focused unit tests pass; an authentic core integration test proves live RW/RO behavior—including the documented nested-mount limit—against an operator-mounted test source using the configured engine root; cancellation leaves no active exec mount and never removes payload. Generated SDKs are intentionally stale at this internal checkpoint and must be completed in chunk 2 before the branch is acceptance-ready.

Chunk 2 — complete user surfaces and acceptance

Meaningful increment: users can pass the volume by SDK and CLI address, modules can consume but not construct it, and generated public surfaces are complete.

  • core/schema/address.go, address tests, and internal/cmd/dagger/flags*: canonical engine-volume address parsing and CLI Volume argument behavior.
  • Introspection/module filtering tests and test module fixtures: constructor hidden and runtime-denied; received volume usable.
  • core/integration/volume_test.go: local/dev-engine setup, grouped names, subdir semantics, remote-engine meaning, nested mount visibility, shared mutation, cache-hit versus cache-busted visibility, invalid/missing/symlink cases, concurrent reuse, disconnect/cancel, and unsupported cases where authentic and practical.
  • Run dagger generate; commit required SDK/schema/reference outputs. Add concise operator-facing docs only if maintainers want them in the feature PR.

Exit criteria: generated diff is clean/reproducible; canonical focused gates pass; DCO and schema/version-gating checks pass; no mount leak is observed. A human checkpoint confirms final constructor/address naming and operator UX before generated output is frozen.

Reuse and review readiness

Both chunks reuse the merged opaque Volume, container dependency, exec mount, module-filtering, address, and integration-test infrastructure. After chunk 2: rebase on fresh upstream/main, review the full generated diff, run only the canonical affected integration/generation gates first, then required CI. 15. Rollout and PR strategy

Use one feature PR unless review shows the server-root interface deserves an independently useful prerequisite. Keep the two chunks as reviewable commits or commit groups, then add generated output with the user surface it reflects. The feature is additive and version-gated, but should initially be documented as Linux-engine/operator-managed functionality. No migration is run, no existing state is moved, and uninstalling an engine does not imply deleting operator volume contents beyond the operator’s normal state-volume policy.

16. Open decisions and recommendations

  1. Public name and scheme. Confirm engineVolume and engine-volume://. Recommendation: use these direct names; alternatives such as “host volume” incorrectly imply the caller host.
  2. Exact portable component grammar and limits. Recommendation: [A-Za-z0-9][A-Za-z0-9._-]*, reserve case-sensitive fs, cap each component at 255 bytes and the canonical name at a conservative 4096 bytes/path-limit-derived bound. The discussion agreed “conservative portable grammar” but not this exact regex/length.
  3. subdir grammar. Recommendation: accept canonical slash-separated relative filesystem components (same traversal bans), but do not reserve fs within payload. This was implied by “subdirectory of the volume,” not explicitly settled.
  4. Constructor timing. Recommendation: validate syntax at construction but create/resolve at exec preparation, preserving lazy Dagger semantics and avoiding a cached object containing a stale path. The discussion required creation when absent but did not decide timing.
  5. Recursive read-only. Recommendation: document that rbind,ro protects the top-level bind but not nested mounts in v1. Requiring recursive read-only would add mount_setattr/rro and kernel/runtime compatibility work for an operator-controlled edge. Erik should confirm the documented-limit choice.
  6. Symlink clamp versus strict rejection. Recommendation: use existing chroot-style safe-root resolution, which guarantees containment but clamps apparent escapes rather than diagnosing them specially. Strict rejection would be new machinery; Erik should confirm that the existing-helper semantics match his intent.

17. Rejected alternatives

  • Arbitrary host path argument: expands the capability beyond operator-curated storage and is ambiguous for remote engines.
  • Flat names only: forces an operator bind mount per logical volume and prevents convenient grouped configuration.
  • No fs payload level: makes administrative hierarchy and payload subdirectories ambiguous and leaves no clean place for future metadata/work directories.
  • Extra Dagger namespaces: adds policy without a broader engine namespace model; users can group names themselves.
  • Snapshot/import as Directory: defeats live mutable access and scales poorly for the motivating trees.
  • Model as CacheVolume: falsely assigns BuildKit snapshot lifecycle, GC, and namespacing semantics.
  • Reuse SSHFS process machinery: introduces secrets, session scoping, FUSE, and teardown that a bind mount does not need.
  • Content-derived caching/watchers: expensive and race-prone for mutable large trees; explicitly deferred.
  • Automatic subdir creation: hides operator/setup mistakes; may be added later behind explicit behavior if demanded.
  • Custom propagation controls in v1: no demonstrated need; recursive bind is the intuitive minimum.
