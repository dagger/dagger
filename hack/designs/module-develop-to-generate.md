# Replace `dagger develop` with `dagger generate`

## Status: Draft

## Prior Art

The blueprint-to-workspace port (`hack/designs/blueprint-to-workspace.md`) follows a similar
pattern: extracting a module-level concept to workspace-level, adding migration for legacy
configs, and cleaning up special-case code paths. Study that design before starting.

## Problem

`dagger develop` creates a parallel universe outside the workspace paradigm. It is a special
command that implicitly means "I'm developing a module, not using one." This distinction:

- **Is confusing** — users must know which commands are "use" commands vs "develop" commands.
- **Is limiting** — adding new dev-time behaviors requires new special commands that
  implicitly say "I'm a develop-time command."
- **Doesn't compose** — `dagger develop` can't benefit from workspace features (multiple
  modules, generators, linting, etc.).

The `dagger develop` command bypasses workspace module loading entirely
(`SkipWorkspaceModules: true`) and directly calls the SDK's `Codegen()` interface. This is a
completely separate code path from the standard workspace-based `dagger generate`.

Meanwhile, `dagger init` for modules has a similar issue: template generation is baked into
the init command rather than being a standard generator operation.

## Design

### Core Insight

Developing a module IS using modules — specifically, using SDK/tooling modules against module
source code. The workspace is already the mechanism for "here are the modules I want to use
to work on my source code." A module being developed should have its own workspace that
references development-time modules (SDK, linters, formatters).

`dagger develop` is replaced by `dagger generate`. The SDK's codegen becomes a regular
`+generate` function in a regular module, invoked through the standard workspace/generator
machinery. No special commands, no special interfaces.

### Module Structure

**Before (current):**
```
my-module/
  dagger.json              # name, sdk.source (used for BOTH runtime AND codegen)
  main.go                  # User code
  dagger.gen.go            # Generated by `dagger develop` via SDK's Codegen() interface
```

**After (new model):**
```
my-module/
  dagger.json              # name, sdk.source (runtime ONLY)
  .dagger/
    config.toml            # Module's dev workspace: references SDK generate module
    lock                   # Lock file for dev workspace dependencies
  main.go                  # User code
  dagger.gen.go            # Generated by `dagger generate` via SDK module's +generate function
```

Key differences:
- `dagger.json` `sdk.source` narrows to runtime-only purpose.
- `.dagger/config.toml` appears inside the module directory, making it a workspace.
- The SDK generate module is referenced as a regular workspace module in `config.toml`.
- Workspace detection stops at the innermost `.dagger/` — no parent workspace leaking.

**Every module has a workspace.** `dagger module init` always creates both `dagger.json` and
`.dagger/config.toml`. The workspace may be empty (Dang) or populated (Go, Python, etc.), but
it always exists. This ensures:
- Workspace detection is always explicit.
- The `-m` flag works consistently.
- Dev tools can be added later without restructuring.

**Example `config.toml` for a Go module:**
```toml
[modules.go-sdk]
source = "github.com/dagger/dagger/sdk/go-dev"
```

**Example with additional dev tooling:**
```toml
[modules.go-sdk]
source = "github.com/dagger/dagger/sdk/go-dev"

[modules.golint]
source = "github.com/dagger/dagger/toolchains/golint"
```

**Dang module (empty workspace, no generate needed):**
```
my-dang-module/
  dagger.json              # sdk.source = "dang"
  .dagger/
    config.toml            # Empty — no generate modules needed, but workspace exists
  main.dang
```

### Decoupled Runtime and Generate

`dagger.json` `sdk.source` declares the runtime: how to execute the module. The dev
workspace's SDK module declares the generate: how to produce generated code. These are
independent concerns that CAN differ:

| Scenario | `dagger.json` sdk.source | `.dagger/config.toml` module |
|---|---|---|
| Standard Go module | `go` | Go SDK generate module |
| Dang module | `dang` | *(empty workspace)* |
| Dockerfile-to-module | `go` | Custom Dockerfile-to-module generator |
| Alternative templates | `go` | Alternative Go SDK with different templates |

This decoupling means:
- SDKs not needing a generate phase (Dang) just have an empty workspace.
- Multiple generate module variants can exist (different templates, strategies).
- The runtime and generate can be completely different modules.

### SDK Generate Module Interface

The SDK generate module is a **regular module** with `+generate` functions. No special SDK
interfaces — it's just a module that knows how to generate code for a specific language.

**What the generate function does:**
1. Reads `dagger.json` from the workspace source to understand the module (name,
   dependencies, etc.).
2. Calls the Dagger API for schema introspection (to generate type bindings).
3. Returns a `Changeset` with all generated files (bindings, `go.mod`/`go.sum`
   adjustments, etc.).

**First-time behavior (templates):** The same generate function handles scaffolding. If it
detects that no user source code exists yet (e.g., no `main.go`), it generates the
boilerplate template too. This unifies what was previously split between `dagger init`
(templates) and `dagger develop` (codegen).

**Client generation:** `ClientGenerator` also becomes a `+generate` function on the SDK
module, configured via workspace config. If no client is configured, no client code is
generated. Everything flows through the standard `+generate` mechanism.

**What this replaces:**

| Current | New model |
|---|---|
| `CodeGenerator.Codegen()` | `+generate` function on SDK module |
| Template generation in `dagger init` | `+generate` detects missing source, generates templates |
| `ClientGenerator.GenerateClient()` | `+generate` function, configured via workspace config |
| `Runtime.Runtime()` | **Unchanged** — stays in `dagger.json` `sdk.source` |

### Bootstrap Chain

**The fundamental constraint:** To load a module, the engine needs a runtime. To generate a
module's code, the generate module must itself be loadable. Circular dependency must be
broken.

**Solution — Dang as the foundation:**

Dang is a native language that doesn't need code generation. Its runtime is baked into the
engine. This breaks the circular dependency.

```
Layer 0: Dang runtime (baked into engine, no codegen needed)
   |
   +-> Layer 1: Go SDK generate module (written in Dang, real module)
   |     Go modules can now be generated
   |
   +-> Layer 1: Go SDK runtime (baked into engine today, could become module later)
   |     Go modules can now be executed
   |
   +-> Layer 2: Python/TS/etc SDK generate modules (written in Go, Dang, or anything)
         All other modules can now be generated
```

**What's baked into the engine:**
- Dang runtime — the irreducible minimum, a language that needs no codegen.
- Go runtime — today. Could potentially be externalized later, but not a goal of this
  design.

**What becomes a real module:**
- Go SDK generate module (written in Dang).
- Python SDK generate module (written in Go or Dang).
- TypeScript SDK generate module (written in Go or Dang).
- Any future SDK generate module.

SDK generate modules are regular, publishable, versionable modules. Different teams can
create alternative generate modules without touching the engine.

### Command Changes

**`dagger init`** — Initializes a workspace (unchanged in purpose).
- Creates `.dagger/config.toml`.
- For projects that want to *use* modules.

**`dagger module init --sdk go`** — Initializes a module for development.
- Creates `dagger.json` with `sdk.source` (runtime).
- Creates `.dagger/config.toml` referencing the SDK generate module.
- Runs `dagger generate` to bootstrap (templates, scaffolding, codegen) — the SDK generate
  function handles everything, including first-time template generation.

**`dagger generate`** — Runs generators from the current workspace (unchanged in purpose,
expanded scope).
- When inside a module's workspace: finds the SDK generate module, runs its `+generate`
  function.
- When inside a project workspace: runs project-level generators as today.
- Replaces `dagger develop` entirely.

**`dagger develop`** — Removed. Clean break, `dagger migrate` handles migration.

**`dagger call`, `dagger check`** — Behavior depends on which workspace you're in.
- Inside a module's dev workspace: operates against the module's dev workspace modules.
- Inside a project workspace: operates against the project's workspace modules.

### Migration

**`dagger migrate` handles existing modules:**

1. Detects modules with `sdk.source` in `dagger.json` but no `.dagger/config.toml`.
2. Creates `.dagger/config.toml` referencing the appropriate SDK generate module:
   - `sdk.source = "go"` -> `github.com/dagger/dagger/sdk/go-dev`
   - `sdk.source = "python"` -> Python SDK generate module
   - `sdk.source = "typescript"` -> TypeScript SDK generate module
   - Custom SDK sources -> best-effort mapping or user guidance.
3. `dagger.json` `sdk.source` remains unchanged (still needed for runtime).
4. Existing generated files remain valid — no forced regeneration.

### What Gets Removed

- `dagger develop` command (`cmd/dagger/module.go`).
- `CodeGenerator` interface from the SDK contract (`core/sdk.go`).
- `ClientGenerator` interface from the SDK contract (`core/sdk.go`).
- `codeGeneratorModule` implementation (`core/sdk/module_code_generator.go`).
- The `SkipWorkspaceModules` path in the develop command.
- Template generation logic from `dagger init` / `dagger module init` (moved to SDK
  generate function).

### What Stays

- `Runtime` interface — still needed, still referenced via `dagger.json` `sdk.source`.
- `ModuleTypes` interface — still needed for self-calls.
- Go SDK runtime baked into engine — unchanged.
- `dagger.json` format — `sdk.source` field narrows in meaning but doesn't change shape.
- Built-in SDK name resolution for runtime (`"go"`, `"python"`, `"typescript"` magic strings
  in `dagger.json` `sdk.source` for runtime resolution).

## Open Questions

- **Go SDK runtime externalization:** Should the Go runtime eventually become a real module
  (written in Dang), so only Dang is baked into the engine? Not a goal of this design, but a
  natural follow-up.
- **SDK generate module naming convention:** What are the actual module paths? `go-dev`,
  `go-sdk-gen`, or something else?
- **Workspace config for client generation:** What does the config surface look like for
  enabling/disabling client generation via workspace config?
