1. Why we're doing this (short)

Every Dagger client gets a private store on the engine holding its telemetry (spans, logs, metrics) so the CLI can stream it out, resume after a disconnect, and drain the rest at shutdown. Today that store is a SQLite database per client.

SQLite allows one writer at a time per database file. But we point many concurrent writers at each file: the client's own telemetry, a copy of every descendant client's telemetry (so the CLI can drain the whole tree from one place), and the HTTP handlers that receive telemetry from processes inside containers. The verified post-fix shutdown flake (trace e791c309…) showed exactly this: during one session's teardown, writes of 4–35 rows spent 1.3–3.2 seconds queued for the main client DB's single write connection (beginWait, up to 111 waiters), while the actual INSERT time was milliseconds. Seven clients flushing at once each accumulated 8–10s of queueing and blew the CLI's 10s shutdown budget. Four rounds of tuning (connection pools, batching, flush scoping, write splitting) each moved this tail without removing it, because one-writer-per-file is the floor of the current design.

Meanwhile, the feature list we actually use from SQLite is very short (§2.6). The proposal: replace the storage layer under the existing clientdb package API with a small append-only log we write ourselves, designed for exactly this access pattern.

2. The hole: what the current system does

If we cut SQLite out, the replacement must fill this exact shape. Every item below was verified against current code (file references inline).

2.1 The data

Three independent streams per client, each a numbered sequence of immutable rows (engine/clientdb/models.go):

type Span struct {          // ~21 fields; the big ones are []byte blobs:
    ID int64                //   Attributes, Events, Links, InstrumentationScope,
    TraceID, SpanID string  //   Resource — each protojson-encoded
    ParentSpanID sql.NullString
    StartTime int64; EndTime sql.NullInt64
    ... }
type Log struct {  ID int64; TraceID, SpanID sql.NullString; Timestamp int64
                   Body, Attributes, InstrumentationScope, Resource []byte; ... }
type Metric struct { ID int64; Data []byte }   // one opaque blob per row

Facts that matter for the design:

  • Rows are never updated or deleted. Cleanup deletes the whole store (DBs.GC). There are no secondary indexes and no triggers (schema.sql).
  • The same span appears as multiple rows: a "live" snapshot when it starts, another when it ends (plus heartbeat re-exports). This is by design — the schema says so — and consumers dedupe by span ID downstream. A typical row is 0.5–3 KiB; the blobs dominate.
  • IDs are per-stream, start at 1, and strictly increase in write order (SQLite AUTOINCREMENT). Gaps are harmless (readers only compare and order). IDs must keep increasing across a close and reopen of the same client's store — reopening today continues the AUTOINCREMENT sequence, and resumed SSE cursors depend on IDs never going backwards.

2.2 The writers

All writes are batch appends. Per destination store, the concurrent writers are (engine/server/session.go:840-940, engine/server/telemetry.go):

  1. The client's own OTel processors — spans (batches up to 2048, every 100ms tick or on flush), logs (batches up to 128, every 250ms), metrics (one row per 5s).
  2. One extra set of the same processors per descendant client. A nested client's providers export both to its own store and to every ancestor's store. This is the fan-in: a session's main store receives writes from every client in the tree.
  3. The OTLP HTTP handlers (/v1/traces etc.) — telemetry POSTed by processes inside containers is written to the posting client's store and every ancestor's store, concurrently (an errgroup per POST).

So a busy session's main store has dozens of goroutines writing small batches at once. That is the workload that must be fast.

2.3 The readers

There are exactly two read paths in the whole codebase:

(a) The SSE drain (engine/server/telemetry.go:695-772) — one loop per stream per subscriber. The CLI subscribes to spans, logs, and metrics; each loop does:

for {
    rows = SelectXSince(cursor, limit=1000)   // WHERE id > cursor ORDER BY id LIMIT 1000
    if empty {
        if terminating { return }              // second empty read after shutdown → done
        wait 100ms | shutdown signal → terminating = true | request gone → return
        continue
    }
    cursor = last row's ID; emit rows as one SSE event (ID = cursor)
}

Semantics the replacement must preserve exactly:

  • Cursor reads: "give me up to N rows with ID > X, in ID order." Nothing else.
  • Resume: after a disconnect the CLI reconnects with its last event ID (X-Last-Event-ID) and continues from there. Cursors may be arbitrarily far behind the newest data.
  • The shutdown handshake: when the client's shutdown begins, the loop switches to "terminating" and exits on the next empty read. Combined with the flush guarantee below, this is how the tail of a trace reliably reaches the CLI.
  • Multiple subscribers to the same client's streams are allowed (each has its own cursor).

(b) The LLM log capture (core/mcp.go:1441-1520) — runs mid-session, concurrent with writes, when an LLM asks for the logs under a span:

  • SelectLogsBeneathSpan(spanID, cursor, limit): find all descendant spans of spanID (recursive walk over parent_span_id), then return log rows whose span is in that set, with ID > cursor, in ID order, up to a limit. Paginated by the caller.
  • SelectSpan(traceID, spanID): fetch one span row to inspect its attributes. Quirk, verified: the query has no ORDER BY, and spans have duplicate rows — so today it returns the first row for that span (the start snapshot), and the attributes it reads are set at span start. The replacement should keep first-row semantics.

A third query (SelectLogsTimespan) exists but has no callers. Dead code; not part of the hole.

2.4 The visibility rule (the one everything leans on)

When a write call returns, the rows must be readable by every reader. Today one SQLite transaction per batch provides this. The shutdown path depends on it: the drain phase force-flushes the OTel processors, whose exporters call the write path; when the flush returns, the SSE terminating drain must be able to see everything, or the trace tail is silently lost. This "returned means readable" rule is the single most load-bearing behavior in the system.

One refinement is allowed: today a whole batch becomes visible atomically (one transaction). Both readers only ever drain forward with cursors, so making rows visible one-append-at-a-time in ID order is indistinguishable to them — they'd just occasionally see data slightly sooner. Nothing observes "batch boundaries." (Checked both read paths.)

2.5 Lifecycle, files, and cleanup

From engine/clientdb/dbs.go and engine/server/session.go:

  • Refcounted opens. DBs.Open(clientID) returns a shared handle and bumps a refcount; Close() decrements; the store fully closes at zero. Each client holds a keep-alive handle for its lifetime; every export batch and SSE loop briefly opens its own. (Pinned by dbs_test.go.)
  • The file outlives the process's interest in it. Closing releases resources, but the on-disk data stays. Reopening the same client ID continues where it left off — including the ID sequence.
  • GC: a sweep every minute deletes stores whose files haven't been modified for 1 hour, skipping clients that are still active or open (DBs.GC(keep), server.go:gcClientDBs).
  • No durability. synchronous=OFF today: a crash may lose anything. Telemetry does not need to survive an engine crash — sessions don't either.

3. The replacement: an append-only log store

3.1 Shape and seam

Keep the clientdb package as the boundary and re-implement its internals. The public surface that survives unchanged: DBs.Open / GC, DB.Close / Read(), the row structs, the five live query methods, and the PB conversion helpers (Span.ReadOnly(), LogsToPB, MetricsToPB). Callers that go away: BeginTx / WithTx / Stats / Begin (the transaction machinery in the two exporters — their insert loops become one Append call; Begin() is already dead code). The mocks in core/telemetry_test.go and core/schema/test_server_test.go keep working because the type names don't change.

Per client store = three streams (spans, logs, metrics). A stream is: an in-memory tail of recent rows, a spill file holding everything older, and a couple of counters.

type stream[Row any] struct {
    mu       sync.Mutex
    nextID   int64      // next ID to assign
    tail     []Row      // rows (tailBase..nextID-1], newest data, always in memory
    tailBase int64      // ID of tail[0]; rows < tailBase live in the spill file
    tailBytes int64     // accounted size of tail

    spill    spillFile  // append-only file + sparse index (§3.4)
}

3.2 Writing

// Append assigns IDs and makes rows visible. When it returns, every
// reader can see them — same guarantee the SQLite transaction gave.
func (s *stream[Row]) Append(rows []Row, size int64) (last int64) {
    s.mu.Lock()
    s.waitBelowHardCap(size)  // cond.Wait releases mu while the spiller drains
    for i := range rows { setID(&rows[i], s.nextID); s.nextID++ }
    s.tail = append(s.tail, rows...)
    s.tailBytes += size
    last = s.nextID - 1
    needSpill := s.tailBytes > tailBudget
    s.mu.Unlock()
    if needSpill { s.requestSpill() }   // wakes the background spiller
    return last
}

The critical section is an ID assignment and a slice append — nanoseconds to microseconds, with no I/O. Below the hard cap, a hundred goroutines appending small batches interleave with negligible waiting, because nobody holds the lock long enough to queue behind. If a stream tail reaches its 64 MiB hard cap, Append waits on the condition variable until the spiller drains enough data (and an oversized batch waits until it is physically spilled). The beginWait convoy has no equivalent on the normal path — there is no transaction to begin and no connection to acquire.

The exporters simplify accordingly: the row-building code they already have (ExportSpans building InsertSpanParams) stays, and the begin/insert-loop/commit block is replaced by one Append. The per-write instrumentation keeps the same log line with appendDuration and spillLag fields replacing the transaction timings (noted in §5 for log comparability).

3.3 Reading

func (s *stream[Row]) Since(id int64, limit int) ([]Row, error) {
    s.mu.Lock()
    if id+1 >= s.tailBase {                    // common case: reader is at the tail
        rows := copyRange(s.tail, id, limit)   // copy row headers out; blobs are
        s.mu.Unlock()                          // immutable []byte, shared safely
        return rows, nil
    }
    base := s.tailBase                          // reader is behind: read the file,
    s.mu.Unlock()                               // NOT holding the lock
    return s.spill.readSince(id, min(limit, base-id-1))  // then tail on next call
}

SSE loops poll at 100ms and are almost always at the tail, so the hot read path is a mutex-guarded slice copy. Readers that fall behind (or resume after reconnect) read from the spill file without holding the stream lock — a behind reader can never slow a writer or another reader. The 100ms polling loop in sseHandler is kept exactly as-is for this design (zero behavior change); switching it to blocking wakeups is a possible follow-up, not part of this change.

3.4 The spill file: how memory stays bounded

The tail is a small window, not the history. A background spiller per stream moves the oldest tail rows into the file whenever the tail exceeds its budget (a few MiB):

  1. Take the lock briefly to grab a batch of the oldest tail rows (they're immutable — safe to encode outside the lock).
  2. Encode and append them to the file (buffered writes, no fsync ever — matching today's synchronous=OFF).
  3. Take the lock again to advance tailBase and drop the spilled rows from memory. Rows leave memory only after they are in the file — so any row is always readable from one place or the other, and the visibility rule survives.

File format, one file per stream (<clientID>.spans.log etc., same directory and GC sweep as today): a small header with a format version, then frames of varint length + row encoding. Row encoding is a hand-written ~30-line field-by-field binary codec per row type (varint ints, length-prefixed strings/bytes) — no reflection, no new dependencies. (Considered: encoding/gob — reflection cost and self-describing overhead per stream; a protobuf mirror of the row structs — codegen churn for three flat structs. Hand-rolled is ~100 lines total including decode and is the fastest of the three; the format is private to the engine and versioned by the header byte.)

To make behind-readers cheap, the spiller keeps a sparse index in memory: every K rows (say 256) it records (first ID, file offset). A read from ID X binary-searches that index, seeks, and scans at most K frames. The index costs 16 bytes per K rows — negligible.

3.5 The two LLM-capture queries

Today these are full-table scans (no indexes exist). The replacement maintains two small maps on the span stream, updated during Append and rebuilt on reopen:

firstRow map[spanID]int64      // span ID → row ID of its FIRST row (matches today's
                               //   no-ORDER-BY first-row semantics for SelectSpan)
children map[spanID][]spanID   // parent → unique child span IDs (for the descendant walk)
  • SelectSpan: map lookup → row ID → read from tail or file. (Faster than today's scan.)
  • SelectLogsBeneathSpan: walk children from the target span to build the descendant set, then cursor-scan the log stream from the caller's cursor, keeping rows whose span ID is in the set, up to the limit. Same worst case as the SQL (it also scans logs forward from the cursor), minus the SQL machinery.

Cost: two map entries per unique span. At 100k unique spans that's on the order of 10–15 MB per client — real, bounded, and only for clients that big; called out in §5.

3.6 Lifecycle mapping

Today (SQLite)Replacement
Open = refcounted handle; pools + prepared statements on first open Same registry, refcounts, and per-client locking (DBs keeps its shape); first open loads or creates the three stream files
Reopen continues AUTOINCREMENT Reopen scans each file forward once to rebuild nextID, the sparse index, and the two span maps (§5 discusses the cost)
Close at refcount 0 releases pools; file stays Close at refcount 0 flushes the spill buffer, drops the tail (after spilling it), stops the spiller; files stay
GC deletes <id>.db* files (mtime > 1h, not active, not open) Identical sweep over <id>.*.log files; during a version transition the sweep also collects leftover .db files
Crash: lose anything (synchronous=OFF)Crash: lose anything (no fsync). Same contract.

No data migration is needed: stores only matter within a session, sessions don't survive an engine upgrade, and stale files age out of the 1-hour GC on their own.

3.7 The shutdown path, after

Flushing a client's telemetry still means "drain the OTel SDK processors through the exporters" — but the exporter's write is now a memory append. There is nothing downstream to wait for: no lock queue, no transaction, no file I/O on the flush path (spilling is the spiller's job and irrelevant to visibility). The same applies to the OTLP POST handlers — the other measured slow-fan-out source — and SSE reads stop competing with writes entirely.

4. Why this is faster, with numbers

OperationToday (measured)Replacement (expected)
Append a batch, uncontended~1ms/row (SQL VM + B-tree + WAL)~1–5µs/row (mutex + slice append; blobs already built by the exporter)
Append a batch, 10s of concurrent writersseconds of beginWait (convoy); the flakeunchanged µs — the lock is held for ns–µs, queues can't form at these hold times
Flush-at-shutdown per clientp99 9s in the failing captureSDK queue drain + appends: milliseconds
SSE tail read (the 10Hz poll)SQL query on the read poolmutex + slice copy
Resumed/behind readerB-tree range scanindexed file seek + sequential decode (page-cache-hot for recent data)
LLM capture queriesfull-table scans (no indexes)map lookup / walk + one cursor scan
MemorySQLite page cache lives in the Go heap (modernc is pure Go), per open DBtail budget (few MiB/stream) + span maps; file cache is OS page cache, reclaimable

The honest core of the argument: the failure we're fixing is lock queueing, and this design removes the queue by making the critical section five orders of magnitude shorter, not by tuning it. Host pressure (the CI runners are overloaded during these failures) still slows everything — but a µs critical section stretched 100× is still 100µs, while a 1ms+lock-queue write stretched 100× is the 10s we measured.

5. Surprises, risks, and open decisions

Found while re-deriving the contract; none is a blocker, all should be decided consciously.

Review outcome (Erik, 2026-07-22): all eight points accepted as written. Specifically: (1) reopen-scan cost accepted for v1, mitigate later only if it bites; (2) stale-store replay behavior is preserved exactly — the underlying question of whether it's desirable may be looked at separately, out of scope here; (3) the clientdb-dump tool ships with phase 2; (4) span-map heap accepted — visible and bounded beats hidden; (5) per-append visibility accepted; (6) the new error shape accepted; (7) the in-flight fan-in writer merges first as an interim fix and is later replaced wholesale by this; (8) scope boundaries confirmed. Proceeding to implementation (phased per §6, review council per phase). This doc lives in the repo at internal-docs/telemetry-store.html.
Round-2 council decision (2026-07-22): keep a 4 MiB spill target per stream and add a 64 MiB blocking hard cap. This leaves measured workloads on the microsecond append path while propagating pathological disk pressure into the bounded OTel BSP queues instead of allowing unbounded heap growth; a wedged disk may block indefinitely, matching the accepted SQLite failure mode.
  1. Reopen cost. Rebuilding nextID, the sparse index, and the span maps on reopen means one sequential read of the files. Full closes mid-session are rare (each client holds a keep-alive handle for its lifetime), but a reconnect after idle would pay it. A ~100MB store scans in well under a second; acceptable, but it's a new cost where SQLite had none. Mitigation if ever needed: periodic checkpoint of the counters into a sidecar file; not in v1.
  2. Stale-store replay is existing behavior we'd preserve. Nested clients have stable (digest-derived) IDs. If the same client ID runs again within the 1-hour GC window, today it reopens the old SQLite file, continues its IDs, and a fresh subscriber (cursor 0) would re-drain the previous run's rows. The replacement reproduces this exactly. Whether that behavior is desirable is a separate question — flagged, not changed here.
  3. We lose sqlite3 as a debugging tool. Real ergonomic loss — being able to query a client DB by hand has been useful. Mitigation: a ~50-line hack/clientdb-dump that decodes a store to JSON lines; the format is trivial. Should ship in the same PR.
  4. New heap where SQLite used its page cache: the span maps. ~10–15MB per 100k-span client (§3.5). Offset by dropping modernc's in-heap page cache, but it's per-client heap that didn't exist as Go objects before. Bounded and measurable; the validation plan includes a heap check at the usual profile points.
  5. Batch visibility becomes per-append instead of per-transaction. Argued harmless in §2.4 (cursor readers can't observe it). Called out so nobody is surprised reviewing.
  6. Error surface shrinks but changes shape. Appends can no longer fail per-row (no constraint checks); the only failure modes are file-system errors in the spiller, which are reported on the next Append/flush of that stream (and logged). The log exporter's current "log and continue on row error" behavior maps to: encoding errors are impossible for rows the exporter already built (it marshaled the blobs itself), I/O errors are stream-fatal-and-logged. Reviewers should check this matches operational expectations.
  7. Relationship to the in-flight fan-in writer work. The per-DB single-writer agent currently being built attacks the same convoy at the SQLite layer. If this design proceeds, that work is an interim fix that this replaces wholesale (its queue is conceptually this design's tail). Sequencing decision — ship fan-in first for the next release and then swap, or leapfrog straight to this — belongs to Erik/vito, not this doc.
  8. What this deliberately does not change: the SSE wire protocol and cursor format, the 100ms poll cadence, what data is stored (including duplicate span rows and the fat per-row resource/scope blobs — vito's row-payload normalization idea is compatible but separate), the OTel processor stack in front, and anything CLI- or Cloud-side.

6. Implementation and test plan

  1. Phase 1 — the store, standalone (new files in engine/clientdb; no callers change): stream, spill file + sparse index, codecs, span maps, DBs registry rework. Tests: port dbs_test.go (refcount lifecycle pins); property tests — append/read equivalence vs a naive in-memory reference model under randomized concurrent writers/readers/spills (with -race); visibility test (Append returned ⇒ Since sees it, hammered concurrently); reopen test (IDs continue, maps rebuilt, behind-cursor reads correct); GC test. Rough size: ~600 lines code, ~600 lines tests.
  2. Phase 2 — swap the callers: exporters lose their transaction blocks, SSE fetchers and mcp queries call the same-named methods, instrumentation fields updated (appendDuration/spillLag; keep the log line name so before/after CI captures stay comparable), hack/clientdb-dump. Gates: focused unit suites, then dagger call engine-dev test --pkg ./dagql/idtui --run='TestTelemetry' (the golden suite covers the whole pipeline end to end, including nested-client span delivery after Close), then a CI-scale capture compared against the flake baseline (/tmp/postfix-check-full.log): expect slow client DB telemetry write count ≈ 0 and the "flush client telemetry" phase p99 in milliseconds.
  3. Rollout: engine-version boundary, no migration (§3.6). Revert path is clean — the package boundary means a revert is one commit.

End of doc. Feedback welcome on: the §5 decisions (especially 2, 6, 7), the tail budget default, and whether the SSE blocking-wakeup follow-up is worth bundling.
